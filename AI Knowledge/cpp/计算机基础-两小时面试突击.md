## 1.1 CPU 缓存层级（Cache Hierarchy）（15 分钟）

### 🔥 面试必考：请描述 CPU 的存储层级结构

```
┌─────────────────────────────────────────┐
│  寄存器 (Register)       1 周期    ~KB   │  ← 最快最小
├─────────────────────────────────────────┤
│  L1 Cache               ~4 周期   32KB  │
├─────────────────────────────────────────┤
│  L2 Cache              ~12 周期  256KB  │
├─────────────────────────────────────────┤
│  L3 Cache              ~40 周期  8-32MB │  ← 多核共享
├─────────────────────────────────────────┤
│  主存 (DRAM)          ~200 周期  GB级   │
├─────────────────────────────────────────┤
│  硬盘/SSD          ~100000 周期  TB级   │  ← 最慢最大
└─────────────────────────────────────────┘
```

### 💡 核心概念

**1. 时间局部性（Temporal Locality）**
> 刚被访问过的数据，短时间内可能再次被访问
```c
for (int i = 0; i < 1000; i++) {
    sum += a[0];  // a[0] 被反复访问 → 缓存命中！
}
```

**2. 空间局部性（Spatial Locality）**
> 刚被访问的数据，其附近的数据很快也会被访问
```c
for (int i = 0; i < 1000; i++) {
    sum += a[i];  // 顺序访问 → 预取到缓存，命中率高！
}
```

**3. Cache Line（缓存行）**
> CPU 不是按字节读内存的，而是一次读一"行"（通常 64 字节）

```c
struct Data {
    int a;      // 4 bytes
    int b;      // 4 bytes (和 a 在同一 cache line 内，很快)
    int c[16];  // 64 bytes (可能跨 cache line)
};
```

### 🔥 和 AI Infra 的关系

在写 CUDA kernel 时，理解 GPU 的缓存层级（L1/Shared Memory/L2/Global Memory）是优化的核心。GPU 的 Shared Memory 本质就是程序员手动管理的 L1 Cache。

### 📝 常见面试追问

| 问题 | 答案要点 |
|------|----------|
| Cache 满了怎么办？ | 替换策略：LRU（最近最少使用）、LFU（最不常用） |
| 什么是 Cache Miss？ | 3C 模型：Compulsory（首次访问必miss）、Capacity（容量不够）、Conflict（地址映射冲突） |
| 多核 CPU 共享哪些 Cache？ | L3 多核共享，L1/L2 每核独有 |

## 1.2 GPU 架构 vs CPU 架构（20 分钟）

### 🔥 面试必考：CPU 和 GPU 的区别是什么？

| 维度 | CPU | GPU |
|------|-----|-----|
| 核心数量 | 少（4-64核） | 多（数千核） |
| 单核性能 | 强（高主频、大缓存） | 弱（低主频、小缓存） |
| 设计目标 | **延迟**优先（单线程快） | **吞吐**优先（总计算量大） |
| 控制逻辑 | 复杂（分支预测、乱序执行） | 简单（顺序执行） |
| 缓存大小 | 大（MB 级） | 小（KB 级） |
| 适合任务 | 复杂控制流、串行任务 | 数据并行、大量相同计算 |

### 💡 一句话总结

> **CPU 是一辆跑车（快但只能载1-2人），GPU 是一辆公交车（慢一点但一次载上百人）。**

### GPU 内存层级（NVIDIA）

```
  越近越快，越小
┌────────────────────────────────────────────────┐
│  寄存器 (Register)        最快  每个线程私有    │
├────────────────────────────────────────────────┤
│  Shared Memory           片上  Block 内共享    │  ← 关键！
├────────────────────────────────────────────────┤
│  L1 Cache                片上                   │
├────────────────────────────────────────────────┤
│  L2 Cache                片外 所有 SM 共享      │
├────────────────────────────────────────────────┤
│  Global Memory (HBM)     最慢  全局共享         │  ← 显存
└────────────────────────────────────────────────┘
```

### 🔥 面试重点：Shared Memory 是什么？

Shared Memory 是 GPU 上的**片上高速缓存**，一个 Block 内的所有线程共享。用它来：

1. **避免重复读 Global Memory**：同一 Block 的线程需要访问相同数据时，一个线程读到 Shared Memory，其他人直接用
2. **线程间通信**：同一 Block 线程可以通过 Shared Memory 交换数据

```cuda
// CUDA 示例：矩阵乘法中用 Shared Memory 优化
__global__ void matmul(float* A, float* B, float* C, int N) {
    __shared__ float tileA[TILE_SIZE][TILE_SIZE];  // Shared Memory
    __shared__ float tileB[TILE_SIZE][TILE_SIZE];
    
    // 每个线程把自己负责的元素从 Global Memory 搬到 Shared Memory
    tileA[threadIdx.y][threadIdx.x] = A[row * N + ...];
    __syncthreads();  // 确保所有线程都搬完了
    
    // 然后从 Shared Memory（快）中计算，而不是直接读 Global Memory（慢）
    for (int k = 0; k < TILE_SIZE; k++) {
        sum += tileA[threadIdx.y][k] * tileB[k][threadIdx.x];
    }
}
```

### 📝 面试会问

| 问题 | 答案 |
|------|------|
| GPU 的 SM 是什么？ | Streaming Multiprocessor，GPU 的计算单元，每个 SM 有几十到上百个 CUDA Core |
| Warp 是什么？ | 32 个线程为一组（NVIDIA），同一 Warp 内线程执行同一条指令（SIMT） |
| Warp Divergence？ | 同一 Warp 内线程走不同分支 → 部分线程被屏蔽 → 性能下降 |
| Bank Conflict？ | Shared Memory 按 bank 组织，多线程同时访问同一 bank → 串行化 → 性能下降 |

> 💡 这部分先有个概念就行，具体写 CUDA 的时候会深入。面试只要能说出"SM、Warp、Shared Memory、Bank Conflict"这几个词就能应付过去了。

## 1.3 进程与线程（15 分钟）

### 🔥 面试必考：进程和线程的区别

| 维度 | 进程（Process） | 线程（Thread） |
|------|----------------|----------------|
| 定义 | 程序的执行实例 | 进程内的执行流 |
| 资源 | 独立的内存空间 | 共享进程的地址空间 |
| 通信 | IPC（管道、消息队列、共享内存） | 直接读写共享变量 |
| 创建开销 | 大（复制地址空间） | 小（共享地址空间） |
| 崩溃影响 | 一个进程崩溃不影响其他 | 一个线程崩溃可能拖垮整个进程 |
| 切换开销 | 大（切换页表、刷新TLB） | 小（只需切换栈和寄存器） |

### 💡 经典比喻

> **进程 = 一栋独立的大楼**（每家有自己的地址、水电、门锁）  
> **线程 = 同一栋楼里的不同房间**（共享水电网，但各有各的房间）

### 🔥 和 AI Infra 的关系

- **多进程**：PyTorch DataLoader 用 `num_workers` 多进程加载数据（绕过 GIL）
- **多线程**：推理服务中处理并发请求，C++ 线程池
- **GPU 中的线程**：一个 CUDA kernel 可能启动百万个线程！

### 上下文切换（Context Switch）

```
线程 A 执行中 → 时钟中断/系统调用 → 保存 A 的寄存器、PC、栈指针
→ 操作系统调度 → 加载 B 的寄存器、PC、栈指针 → 线程 B 执行
```

**开销来源**：保存/恢复寄存器、TLB 刷新、Cache 可能被污染

### 📝 面试追问

| 问题 | 答案 |
|------|------|
| 协程（Coroutine）是什么？ | 用户态的"轻量级线程"，切换不经过内核，开销极小 |
| 线程池是什么？ | 预先创建一组线程，有任务就分配，用完回收，避免反复创建销毁 |

## 1.4 虚拟内存（10 分钟）

### 🔥 面试必考：什么是虚拟内存？为什么需要它？

**虚拟内存**：每个进程都有自己独立的虚拟地址空间，操作系统通过**页表**将虚拟地址映射到物理地址。

```
程序看到的地址（虚拟地址）     实际物理内存地址
     0x00001000         →     0xABCD1000
     0x00002000         →     0x12345000
     0x00003000         →     不在物理内存中 → 缺页中断 → 从磁盘加载
```

### 三大作用

1. **隔离**：进程 A 不能访问进程 B 的内存（安全）
2. **扩大可用内存**：把不常用的内存换出到磁盘（Swap），骗程序说"你有 128G 内存！"
3. **简化编程**：每个程序都以为自己是 0 地址开始的，不用操心物理地址

### 💡 关键概念：缺页中断（Page Fault）

程序访问的虚拟地址不在物理内存 → 触发异常 → OS 从磁盘加载对应页 → 恢复执行

在 AI Infra 中：CPU 内存不足时使用 `swap`，但 GPU 显存不足会直接 OOM（GPU 没有 swap！）

### 🔥 和 AI Infra 的关系

- **CUDA Unified Memory**：CPU 和 GPU 共享虚拟地址空间，OS 自动迁移数据（类似虚拟内存的扩展）
- **大模型推理**：量化、KV Cache 的 offload（把部分数据放到 CPU 内存甚至硬盘）

## 2.1 并发编程基础（20 分钟）

### 🔥 面试必考：什么是线程安全？如何保证？

**线程安全**：多个线程同时访问同一份数据，不会产生错误结果。

### 互斥锁（Mutex）

```cpp
#include <iostream>
#include <thread>
#include <mutex>
using namespace std;

int counter = 0;
mutex mtx;  // 互斥锁

void increment() {
    for (int i = 0; i < 100000; i++) {
        mtx.lock();         // 加锁
        counter++;          // 临界区（Critical Section）
        mtx.unlock();       // 解锁
    }
}

int main() {
    thread t1(increment);
    thread t2(increment);
    t1.join();
    t2.join();
    cout << "counter = " << counter << endl;  // 200000（正确！）
    return 0;
}
```

> 如果不加锁，两个线程同时 `counter++` 可能只加了一次（竞态条件 Race Condition）。

### 💡 `lock_guard`：C++ 的 RAII 锁

手动 `lock()`/`unlock()` 容易忘记解锁（尤其是有异常的时候），C++ 提供了 RAII 风格的锁：

```cpp
void increment() {
    for (int i = 0; i < 100000; i++) {
        lock_guard<mutex> guard(mtx);  // 构造时自动 lock()
        counter++;
    }  // guard 析构时自动 unlock()，即使有异常也会解锁！
}
```

### 🔥 面试必考：死锁（Deadlock）

**四个必要条件**（缺一不可）：
1. **互斥**：资源不能被共享
2. **持有并等待**：持有资源的同时等待其他资源
3. **不可抢占**：不能强行抢走别人的资源
4. **循环等待**：A 等 B，B 等 C，C 等 A

```cpp
// 死锁示例
// 线程1：先锁 mtxA，再锁 mtxB
// 线程2：先锁 mtxB，再锁 mtxA   ← 顺序不一致！死锁！
```

**如何避免**：按统一顺序加锁！用 `std::lock()` 同时锁多个锁。

### 原子操作（Atomic）

```cpp
#include <atomic>
atomic<int> counter(0);  // 原子变量，不需要 mutex！

void increment() {
    for (int i = 0; i < 100000; i++) {
        counter++;  // 原子操作，硬件保证，比 mutex 快得多！
    }
}
```

| 方案 | 性能 | 适用场景 |
|------|------|----------|
| Mutex | 慢（系统调用） | 复杂临界区、多个变量 |
| Atomic | 快（硬件指令） | 简单计数器、flag |
| 无锁编程 | 最快 | 高手专属，容易写错 |

### 🔥 和 AI Infra 的关系

- 多卡训练的梯度同步（AllReduce）需要考虑并发
- 推理服务的并发请求处理
- NCCL 底层大量使用原子操作和锁

## 2.2 编译原理（15 分钟）

### 🔥 面试必考：C/C++ 程序从源码到可执行文件经历哪些阶段？

```
main.cpp → 预处理 → 编译 → 汇编 → 链接 → a.out
          (Preprocess)  (Compile)  (Assemble)  (Link)
```

### 四阶段详解

**阶段 1：预处理（Preprocessing）**
- 展开 `#include`（把头文件内容复制进来）
- 替换 `#define` 宏
- 条件编译 `#ifdef` `#ifndef`
- 删除注释

```bash
g++ -E main.cpp -o main.i   # 查看预处理结果
```

**阶段 2：编译（Compilation）**
- 把预处理后的 C/C++ 代码翻译成**汇编代码**
- 做语法分析、语义分析、优化

```bash
g++ -S main.i -o main.s     # 查看汇编代码
```

**阶段 3：汇编（Assembly）**
- 把汇编代码翻译成机器码（**目标文件 .o**）
- 此时地址还是"可重定位"的（未确定最终地址）

```bash
g++ -c main.s -o main.o     # 生成目标文件
```

**阶段 4：链接（Linking）**
- 把多个目标文件 + 库文件链接成**可执行文件**
- 解决符号引用（比如你调用了 `printf`，链接器找到 `libc` 中的实现）
- 确定最终地址

```bash
g++ main.o -o a.out         # 链接成可执行文件
```

### 💡 静态链接 vs 动态链接

| | 静态链接 | 动态链接 |
|------|----------|----------|
| 时机 | 编译时 | 运行时 |
| 文件 | `.a`（Linux）`.lib`（Windows） | `.so`（Linux）`.dll`（Windows） |
| 优点 | 不需要外部依赖 | 节省空间，可热更新 |
| 缺点 | 文件体积大 | 依赖找不到就报错 |

### 🔥 和 AI Infra 的关系

- **AI 编译器**（Triton、TVM、XLA）：把高层 IR 编译成 GPU kernel
- **CUDA 编译**：`.cu` → PTX（中间表示）→ SASS（GPU 机器码）
- **JIT 编译**：PyTorch 2.0 的 `torch.compile` 就是在运行时编译

## 2.3 内存管理（15 分钟）

### 🔥 面试必考：堆（Heap）和栈（Stack）的区别

| 维度 | 栈（Stack） | 堆（Heap） |
|------|------------|------------|
| 分配方式 | 编译器自动 | 程序员手动（`new`/`malloc`） |
| 大小 | 小（~8MB） | 大（受物理内存限制） |
| 速度 | 快（移动栈指针即可） | 慢（需要查找空闲块） |
| 生命周期 | 离开作用域自动释放 | 手动释放（`delete`/`free`） |
| 碎片问题 | 无 | 有（需要内存池等方案） |
| 典型存放 | 局部变量、函数参数 | 动态分配的对象、大数组 |

```cpp
void example() {
    int x = 10;              // 栈上分配
    int* p = new int(20);    // p 在栈上，*p 在堆上
    
    // ... 使用 p
    
    delete p;                // 必须手动释放！不然内存泄漏
    // 离开函数后，x 自动释放，p 自动释放，但如果忘了 delete，*p 就泄漏了
}
```

### 💡 内存泄漏（Memory Leak）

堆上分配的内存没有被释放，程序占用内存越来越大。C++ 中：

```cpp
// 传统 C++：容易泄漏
int* p = new int[1000000];
// 忘记 delete[] p; → 泄漏！

// 现代 C++：智能指针，自动管理
unique_ptr<int[]> p = make_unique<int[]>(1000000);
// 离开作用域自动释放！不需要手动 delete
```

### 🔥 内存对齐（Memory Alignment）

CPU 读取数据时有对齐要求。访问未对齐的数据会慢甚至报错。

```cpp
// 考虑这个结构体
struct Bad {
    char c;    // 1 byte
    int i;     // 4 bytes
    char d;    // 1 byte
};
// sizeof(Bad) = ? 不是 6！是 12！

// 实际内存布局（编译器自动插入 padding）
// | c | 0 | 0 | 0 |  i (4 bytes)  | d | 0 | 0 | 0 |
//   ↑      3 bytes padding↑                ↑ 3 bytes padding ↑

struct Good {
    int i;     // 4 bytes (先放大的)
    char c;    // 1 byte
    char d;    // 1 byte
};
// sizeof(Good) = 8 (省了 4 字节！)
```

**经验**：结构体中把大的成员放前面，小的放后面，可以减少 padding 浪费。

### 🔥 和 AI Infra 的关系

- **内存池**：深度学习框架常用，减少 `malloc`/`free` 开销
- **显存管理**：PyTorch 的 Caching Allocator，避免频繁 cudaMalloc/cudaFree
- **内存对齐**：CUDA kernel 中确保数据对齐能加速访问

## 2.4 数据结构快速回顾（10 分钟）

AI Infra 面试中高频出现的几个数据结构：

### unordered_map 的底层

| 容器 | 底层实现 | 查询 | 插入 | 有序？ |
|------|----------|------|------|--------|
| `std::map` | 红黑树 | O(log n) | O(log n) | ✅ |
| `std::unordered_map` | 哈希表 | O(1) 平均 | O(1) 平均 | ❌ |
| Python dict | 哈希表 | O(1) 平均 | O(1) 平均 | ❌（Python 3.7+ 保留插入序） |

### 哈希冲突如何解决？

1. **拉链法**（Chaining）：冲突了就挂链表（C++ 的 `unordered_map`）
2. **开放寻址**（Open Addressing）：冲突了就找下一个空位（Python 的 dict）

当负载因子（元素数/桶数）太高时，触发 **rehash**，重新分配更大的桶数组。

### 一张图总结 GPU 架构

```
GPU
├── SM (Streaming Multiprocessor) × 几十到上百个
│   ├── CUDA Core × 64+
│   ├── 寄存器 × 64K (每个 SM)
│   ├── Shared Memory (程序员可控的 L1)
│   ├── Warp Scheduler (以 32 线程为一组调度)
│   └── L1 Cache
├── L2 Cache (所有 SM 共享)
└── HBM/Global Memory (显存，几百 GB~TB)
```

### 一句诀

| 概念 | 口诀 |
|------|------|
| Cache 局部性 | 刚用过的还会用（时间），旁边的一起用（空间） |
| CPU vs GPU | CPU 跑车快但坐的人少，GPU 公交慢但装得多 |
| 进程 vs 线程 | 进程是独栋大楼，线程是同楼房间 |
| 死锁四条件 | 互斥持有不可抢，循环等待死翘翘 |
| 编译四阶段 | 预处理→编译→汇编→链接（一路往下走） |
| 堆 vs 栈 | 栈是写字楼（小、快、自动），堆是仓库（大、慢、手动） |
